# FLAIR: OSS + API戦略の検討

## 結論

OSSとAPI両方の提供は十分理にかなっている。FLAIRの特性を考えると良い戦略。

## 「OSSあればAPI要らない」は一部のユーザーにしか当てはまらない

FLAIRは確かにシンプル（435行、numpy/scipyのみ、CPU）なので、技術者なら自分で動かせる。でもAPIの価値は「動かせるかどうか」だけじゃない。

### APIが刺さるユーザー層

- **非エンジニアの分析者** — Pythonは書けるけどインフラは触りたくない人
- **プロトタイプ段階のスタートアップ** — forecast APIを1行のHTTPで呼びたい
- **他言語のシステム** — Java/Go/JSのバックエンドからPythonライブラリを呼ぶのは面倒
- **大量バッチ処理** — 数万系列を並列で回すとき、スケーリングを自分で組みたくない
- **SaaS組み込み** — 自社プロダクトにforecast機能を埋め込みたいが、Python依存を入れたくない

## OSSがあることはAPI事業にプラスに働く

よくある不安「OSSだから誰もAPI使わないのでは？」は逆。

1. **OSSが信頼を作る** — 中身が見えるから安心して有料APIを使える。ブラックボックスのAPIより選ばれやすい
2. **OSSがマーケティングになる** — ベンチマーク結果 → GitHub星 → ブログ記事 → API顧客という流れ
3. **成功事例は多い** — Redis, Elasticsearch, Sentry, PostHog, Supabase… 全部OSS+有料サービス

## FLAIRに特に向いている理由

| FLAIRの特性 | APIとの相性 |
|---|---|
| 依存が軽い（numpy/scipy） | デプロイコストが極めて低い→利益率が高い |
| GPUが不要 | サーバー費が安い→価格競争力がある |
| ゼロハイパラ | ユーザーがチューニングで迷わない→APIの体験が良い |
| ベンチマーク1位の実績 | セールストークが明確 |

## 具体的な差別化の方向性

APIでOSSにない価値を乗せる：

- **マネージド実行** — インフラ不要、1 HTTP callで完結
- **バッチ/スケジューリング** — 大量系列の定期予測
- **前後処理の統合** — 欠損補完、異常値検知、結果の可視化
- **SLA/サポート** — 本番利用での安心感
- **マルチテナント対応** — 企業向けのセキュリティ・認証

## 注意点

- 435行・依存2個なので、技術力のあるチームは確実に自前で動かす。**それでいい**。彼らはAPIの顧客じゃない
- 価格設定が重要。「自分で動かす手間」より安ければ使われる。時系列1本あたり数セント程度のレンジ
- 最初はAPIより**Pythonパッケージ（PyPI）の整備**が先かもしれない。`pip install flair-forecast` で使えるだけで採用が増え、API顧客の入口にもなる

## まとめ

OSSは「使える人」に届き、APIは「使いたいけど自分で動かしたくない人」に届く。客層が違うので共存する。
